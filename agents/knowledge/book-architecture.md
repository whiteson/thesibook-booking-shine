# Book Platform Architecture — ThesiBook + Easy!Appointments

## Context

ThesiBook monorepo:

```txt
frontend/     Next.js marketing site + future registration
backend/      WordPress CMS (content only)
book/         Easy!Appointments (scheduling engine, upstream clone)
services/     (future) control plane API for tenant provisioning
```

Easy!Appointments (EA) is **single-tenant by design**: one install, one MySQL database, one admin hierarchy (admin → providers/secretaries → customers).

## Recommended multi-tenant strategy

### Phase 1 (now): Single tenant, learn the product

Run one EA instance locally. Map REST API (`book/openapi.yml`), admin flows, and DB schema. Wire frontend embed/booking links manually.

### Phase 2 (MVP): Database-per-workspace + control plane

**Recommended for ThesiBook** — strong isolation, minimal changes to EA core, aligns with “different database per user.”

```txt
Registration (frontend)
    → Control plane API
        → CREATE DATABASE ea_{workspace_slug}
        → Run EA schema migration / setup
        → CREATE admin user in tenant DB
        → Store workspace record (id, slug, db_name, db_user, status)
    → Return workspace URL: https://{slug}.book.example.com
```

| Layer | Responsibility |
|-------|----------------|
| **Control plane DB** (`thesibook_control`) | Users, workspaces, billing, DB credentials (encrypted), subdomain mapping |
| **Tenant DB** (`ea_{workspace_id}`) | Full EA schema — appointments, services, providers |
| **Routing** | Reverse proxy resolves `{slug}` → tenant config (dynamic `config.php` or env injection) |
| **Frontend** | Registration, dashboard links, iframe or API-driven booking widget |

**Why DB-per-tenant (not shared schema)?**

- EA has no `tenant_id` column anywhere — shared DB requires forking every model/query.
- Per-tenant DB gives clean backup, export, and GDPR deletion.
- Operational cost is manageable up to hundreds of tenants on one MySQL host.
- Matches your instinct; validate ops at scale before optimizing.

### Phase 3 (scale): Harden provisioning

- Connection pooling / read replicas per region
- Automated backups per tenant DB
- Rate limits on provisioning API
- Optional: move heavy tenants to dedicated MySQL instances

### Alternatives (not recommended for MVP)

| Approach | Pros | Cons |
|----------|------|------|
| Shared DB + `tenant_id` | One migration path | Massive EA fork, high regression risk |
| One EA Docker container per tenant | Perfect isolation | Expensive, slow provisioning |
| EA Premium multi-company (if exists) | Vendor-supported | License cost, less control |

## Dynamic config options

EA reads `config.php` class `Config` for `BASE_URL` and DB credentials.

1. **Subdomain proxy + generated config** — PHP-FPM pool per tenant or single entry script that loads tenant config from control plane before bootstrapping EA.
2. **Environment variables** — Patch EA bootstrap to read `getenv('EA_DB_NAME')` etc. (small fork, easier in Docker/K8s).
3. **Path-based** — `book.example.com/w/{slug}/` — requires EA `BASE_URL` and rewrite rules per tenant.

**Default recommendation:** subdomain `{slug}.book.domain` + env-based config injection + single EA codebase.

## Authentication bridge

| Actor | Where they auth | Notes |
|-------|-----------------|-------|
| Platform user (registers on frontend) | ThesiBook auth (Next.js / WP / custom) | Owns workspace |
| Workspace admin | EA backend login | Created during provisioning |
| End customer booking | EA public booking flow | No platform account required |

Do **not** merge EA session cookies with frontend auth in v1. Link via workspace slug + API tokens for server-to-server calls.

## API integration

EA exposes REST API v1 (`/index.php/api/v1/`). Auth: Bearer token or Basic (admin API token from EA settings).

Frontend patterns:

- **Embed** — iframe to tenant public booking URL (fastest MVP).
- **Headless** — Next.js calls EA API for availabilities + appointment creation (more work, better UX).

Swagger UI local: `http://localhost:8092` (when Docker stack includes swagger-ui).

## GPL-3.0 compliance

EA is GPL-3.0. If you distribute modified EA code, provide source. Fork on GitHub, document patches in `book/PATCHES.md`. Control plane in `services/booking/` can remain proprietary if it only communicates with EA via API/network — confirm with legal counsel for your deployment model.

## Security checklist (production)

- Encrypt tenant DB passwords at rest in control plane
- Provisioning API: authenticated, idempotent, audited
- Per-tenant API tokens rotated on demand
- No cross-tenant DB credentials in logs
- Rate limit public booking endpoints per workspace
